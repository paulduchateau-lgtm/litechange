Conventions de nommage

Classes BEM, tiers de tokens, taxonomie des composants et structure de fichiers.

Section A — Classes CSS

BEM-like Naming

Toutes les classes suivent une convention BEM adaptee, prefixee par liteops-. Le composant est en PascalCase, les elements et modificateurs en minuscules.

Pattern: composant
/* Composant de base */ .liteops-Component { ... } /* Variante (modificateur) */ .liteops-Component--variant { ... } /* Element enfant */ .liteops-Component__element { ... }
Pattern: etats
/* Classes d'etat sur le wrapper */ .is-error { ... } .is-disabled { ... } .is-active { ... } .is-hidden { ... } /* L'etat cascade aux enfants */ .liteops-Input.is-error .liteops-Input__field { border-color: var(--liteops-border-error); }
Exemples reels du design system
/* Button */ .liteops-Button /* base */ .liteops-Button--primary /* variante primaire */ .liteops-Button--secondary /* variante secondaire */ .liteops-Button--ghost /* variante ghost */ .liteops-Button--sm /* taille small */ .liteops-Button--lg /* taille large */ /* Badge */ .liteops-Badge /* base */ .liteops-Badge--success /* variante succes */ .liteops-Badge--warning /* variante attention */ .liteops-Badge__dot /* element point colore */ /* Hero Section */ .liteops-Hero /* base */ .liteops-Hero__heading /* element titre */ .liteops-Hero__subheading /* element sous-titre */ .liteops-Hero__actions /* element conteneur boutons */ /* KPI Card */ .liteops-KpiCard /* base */ .liteops-KpiCard__label /* element label */ .liteops-KpiCard__value /* element valeur */ .liteops-KpiCard__trend /* element tendance */ /* Data Table */ .liteops-DataTable /* base */ .liteops-DataTable__header /* element en-tete */ .liteops-DataTable__cell--numeric /* element + modificateur */
Section B — Tiers de tokens

Token Naming

Les tokens CSS sont organises en trois tiers, du brut (valeurs fixes) au semantique (roles d'interface) puis au composant (proprietes specifiques).

Tier Pattern Exemple Usage
Tier 1 -- Raw --liteops-{scale}-{step} --liteops-olive-900 Valeurs brutes de palette. Ne jamais utiliser directement en production.
Tier 2 -- Semantic --liteops-{role}-{variant} --liteops-text-primary Tokens a usage direct. Referencent les Tier 1 et changent selon le theme.
Tier 3 -- Component --liteops-{component}-{property} --liteops-interactive-primary Tokens specifiques a un contexte d'interaction ou de composant.
Exemples par tier
/* Tier 1 — Raw palette (non-themed) */ --liteops-olive-900: #1C1D1A; --liteops-paper-100: #F5F4F0; --liteops-lite-400: #B0D838; --liteops-space-4: 16px; --liteops-radius-md: 10px; /* Tier 2 — Semantic (themed: light/dark) */ --liteops-surface-primary: var(--liteops-paper-100); --liteops-text-primary: var(--liteops-olive-900); --liteops-accent-primary: var(--liteops-lite-700); --liteops-status-success: var(--liteops-success-500); /* Tier 3 — Component (themed: light/dark) */ --liteops-interactive-primary: var(--liteops-lite-700); --liteops-interactive-primary-text: var(--liteops-white); --liteops-chart-primary: var(--liteops-lite-700);

Section D — Structure de fichiers

File Naming

Organisation des fichiers dans le design system. Chaque composant est un fichier HTML standalone dans son tier.

design-system/
  tokens/
    liteops-tokens.css -- CSS custom properties (source of truth)
    tailwind.config.js -- Tailwind theme extension

    atoms/
